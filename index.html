第一次博客搭建

一、初衷：从收藏夹到自留地
最初萌生搭建博客的念头，源于收藏夹里积灰的千篇教程。总以为 “写作” 该在某个完美时刻启动，直到发现：当收藏夹塞满 “如何开始写作” 的文章时，本质上是在用 “准备” 逃避 “开始”。
技术执念：不想用现成平台，偏要自己搭 —— 或许是程序员的 “基建情结”，觉得亲手砌砖的小屋才叫家。
对抗遗忘：笔记软件里散落的思考像断线珠子，而博客像个带橱窗的书架，能把碎片按主题陈列，让知识从 “临时寄存” 变成 “永久展出”。
二、搭建实录：从环境报错到初见雏形

  1. 技术栈选择的纠结症
    方案 优势 最终放弃原因
    WordPress 可视化后台易上手 担心数据库维护复杂度
    Notion 导出 零代码一键生成 自定义空间太有限
    Hexo 静态部署轻量高效 需啃文档（最终选了它！）
  2. 踩坑日记：那些让我凌晨三点睡不着的报错
    Node.js 版本冲突：教程用的 Hexo 6.0，我装了 7.0，结果hexo g命令疯狂报Promise错误，最后发现是某个插件不兼容，降级后秒解决。
    主题配置 “玄学”：选了个 GitHub 上星标很高的主题，文档却只有寥寥几句，为了让侧边栏显示分类，对着主题源码啃了两小时，终于在_config.yml里找到被注释掉的categories: true。
    域名绑定 “玄学 2.0”：阿里云域名解析时，A 记录和 CNAME 搞反了，浏览器输入www.example.com一直跳转到服务器默认页面，最后发现是 DNS 缓存没刷新，喝了杯茶回来突然就通了。
    三、意外收获：比博客更重要的事
  3. 技术与人文的奇妙缝合
    写前端代码时总抱怨 “为什么要调像素级布局”，但给博客调字体行高时，却为了让段落间距从 1.5 倍线改成 1.8 倍而反复预览 —— 当技术服务于表达时，枯燥的配置突然有了温度。
    为了让图片加载更快，研究了 WebP 格式转换和 OSS 存储，第一次觉得 “性能优化” 不是 KPI，而是对读者体验的尊重。
  4. 重新理解 “创作”
    以前觉得写作要等 “灵感爆发”，但搭建博客后发现：当你为文章设置date字段、规划categories时，创作就变成了可管理的日常。现在每周日晚上花 1 小时写草稿，像给数字花园浇水。
    发现 “不完美上线” 的勇气：我的第一篇文章发布时，侧边栏搜索功能还没调好，但想着 “先让内容出来，问题可以慢慢改”，结果反而收到读者关于内容的反馈，比纠结细节更有价值。
    四、给新手的碎碎念
    别追求 “终极方案”：我曾花三天对比 Hexo、Hugo、Jekyll，最后发现随便选一个先开始，比纠结选型更重要 —— 技术只是载体，内容才是灵魂。
    善用 “抄作业” 思维：去 GitHub 搜starred:>1000 blog，克隆别人的配置文件，对着 README 改参数，比啃官方文档效率高 10 倍（但记得注明主题来源）。
    接受 “不被看见” 的常态：我的博客上线半个月，日均访问量 3 人（包括我妈），但每次打开后台看到/archives页面里逐渐变厚的文章列表，就觉得像在攒一本电子版的 “成长手账”。
    结语：数字时代的手写感
    搭建博客最惊喜的发现是：当我在 Markdown 里敲下# 标题、用> 引用标注金句时，这些看似冰冷的符号，最终会在浏览器里变成有温度的文字。就像用代码搭了个壁炉，而每篇文章都是往里面添的柴火 —— 或许在算法洪流里，这样一个能自己决定 “摆什么家具”“挂什么画” 的小空间，本身就是对抗同质化的温柔仪式。
    （附：我的博客地址 www.example.com ，虽然还很简陋，但欢迎来踩脚印～）
